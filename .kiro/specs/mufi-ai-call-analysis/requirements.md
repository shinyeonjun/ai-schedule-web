# 요구사항 명세서 (Requirements Document)

## 소개 (Introduction)

프로젝트는 AI 기반 통화 분석 시스템으로, 통화 내용에서 일정 정보를 자동으로 추출하고 관리하는 웹 애플리케이션입니다. OpenAI GPT-4o-mini를 활용하여 통화 텍스트를 분석하고, Google Calendar 및 Gmail과 연동하여 일정을 자동으로 등록하고 공유할 수 있습니다. 사용자는 Google OAuth 2.0을 통해 안전하게 로그인하며, 분석된 일정을 수정, 저장, 공유할 수 있습니다.

## 요구사항 (Requirements)

### 요구사항 1: 사용자 인증 및 관리

**사용자 스토리:** 사용자로서, Google 계정으로 안전하게 로그인하여 내 일정 데이터를 관리하고 싶습니다.

#### 수락 기준 (Acceptance Criteria)

1. WHEN 사용자가 로그인 페이지에 접속하면 THEN 시스템은 Google OAuth 2.0 로그인 버튼을 표시해야 합니다.
2. WHEN 사용자가 Google 로그인 버튼을 클릭하면 THEN 시스템은 Google 인증 페이지로 리다이렉트해야 합니다.
3. WHEN 사용자가 Google 인증을 완료하면 THEN 시스템은 JWT 토큰을 생성하고 대시보드로 리다이렉트해야 합니다.
4. WHEN 사용자 정보가 데이터베이스에 없으면 THEN 시스템은 새로운 사용자 레코드를 생성해야 합니다.
5. WHEN 사용자가 로그인하면 THEN 시스템은 마지막 로그인 시간을 업데이트해야 합니다.
6. WHEN JWT 토큰이 만료되면 THEN 시스템은 401 에러를 반환하고 재로그인을 요청해야 합니다.
7. WHEN 사용자가 대시보드에 접근하면 THEN 시스템은 JWT 토큰을 검증하고 사용자 정보를 표시해야 합니다.

### 요구사항 2: 통화 내용 분석

**사용자 스토리:** 사용자로서, 통화 내용을 입력하거나 업로드하여 AI가 자동으로 일정 정보를 추출하도록 하고 싶습니다.

#### 수락 기준 (Acceptance Criteria)

1. WHEN 사용자가 텍스트 파일(.txt)을 업로드하면 THEN 시스템은 파일 내용을 읽고 분석을 시작해야 합니다.
2. WHEN 업로드된 파일이 10MB를 초과하면 THEN 시스템은 400 에러를 반환하고 파일 크기 제한 메시지를 표시해야 합니다.
3. WHEN 사용자가 텍스트를 직접 입력하면 THEN 시스템은 입력된 내용을 분석해야 합니다.
4. WHEN 분석이 시작되면 THEN 시스템은 OpenAI GPT-4o-mini API를 호출하여 일정 정보를 추출해야 합니다.
5. WHEN AI 분석이 완료되면 THEN 시스템은 그룹 일정(group)과 개인 일정(personal)으로 분류된 결과를 반환해야 합니다.
6. WHEN 일정 정보가 추출되면 THEN 각 일정은 제목, 설명, 장소, 시작 시간, 종료 시간, 참석자 정보를 포함해야 합니다.
7. WHEN 분석 중 오류가 발생하면 THEN 시스템은 500 에러를 반환하고 오류 메시지를 표시해야 합니다.

### 요구사항 3: 일정 데이터 관리

**사용자 스토리:** 사용자로서, 분석된 일정을 수정하고 데이터베이스에 저장하여 나중에 다시 확인하고 싶습니다.

#### 수락 기준 (Acceptance Criteria)

1. WHEN 사용자가 분석된 일정을 저장하면 THEN 시스템은 Supabase 데이터베이스에 일정 데이터를 저장해야 합니다.
2. WHEN 일정이 저장되면 THEN 시스템은 고유한 분석 세션 ID(analysis_session_id)를 생성해야 합니다.
3. WHEN 사용자가 일정 목록을 조회하면 THEN 시스템은 해당 사용자의 모든 분석 세션을 최신순으로 반환해야 합니다.
4. WHEN 사용자가 특정 세션을 선택하면 THEN 시스템은 해당 세션의 모든 일정을 그룹/개인으로 분류하여 반환해야 합니다.
5. WHEN 사용자가 일정의 필드를 수정하면 THEN 시스템은 데이터베이스에서 해당 일정을 업데이트해야 합니다.
6. WHEN 사용자가 일정을 삭제하면 THEN 시스템은 데이터베이스에서 해당 일정을 삭제해야 합니다.
7. WHEN 사용자가 분석 세션을 삭제하면 THEN 시스템은 해당 세션의 모든 일정을 삭제해야 합니다.
8. WHEN 사용자가 세션 제목을 수정하면 THEN 시스템은 해당 세션의 모든 일정의 analysis_source_name을 업데이트해야 합니다.
9. WHEN 다른 사용자의 일정에 접근하려고 하면 THEN 시스템은 404 에러를 반환하고 접근을 거부해야 합니다.

### 요구사항 4: Google Calendar 연동

**사용자 스토리:** 사용자로서, 분석된 일정을 Google Calendar에 자동으로 추가하여 내 캘린더에서 확인하고 싶습니다.

#### 수락 기준 (Acceptance Criteria)

1. WHEN 사용자가 일정을 Google Calendar에 추가하면 THEN 시스템은 Google Calendar API를 호출해야 합니다.
2. WHEN Google Calendar API 호출이 성공하면 THEN 시스템은 생성된 캘린더 이벤트 ID와 URL을 반환해야 합니다.
3. WHEN Google 토큰이 만료되었으면 THEN 시스템은 자동으로 토큰을 갱신해야 합니다.
4. WHEN 토큰 갱신이 실패하면 THEN 시스템은 사용자에게 재인증을 요청해야 합니다.
5. WHEN 캘린더 이벤트가 생성되면 THEN 이벤트는 일정의 제목, 설명, 장소, 시작/종료 시간, 참석자 정보를 포함해야 합니다.
6. WHEN Google Calendar 권한이 없으면 THEN 시스템은 500 에러를 반환하고 권한 확인 메시지를 표시해야 합니다.

### 요구사항 5: Gmail 이메일 발송

**사용자 스토리:** 사용자로서, 일정 정보를 참석자들에게 이메일로 자동 발송하여 일정을 공유하고 싶습니다.

#### 수락 기준 (Acceptance Criteria)

1. WHEN 사용자가 일정 이메일 발송을 요청하면 THEN 시스템은 Gmail API를 사용하여 이메일을 발송해야 합니다.
2. WHEN 이메일이 발송되면 THEN 시스템은 ICS 파일을 첨부하여 수신자가 캘린더에 추가할 수 있도록 해야 합니다.
3. WHEN 여러 수신자가 지정되면 THEN 시스템은 각 수신자에게 개별적으로 이메일을 발송해야 합니다.
4. WHEN 이메일 본문이 생성되면 THEN 일정의 제목, 시간, 장소, 설명이 HTML 형식으로 포함되어야 합니다.
5. WHEN Gmail 권한이 없으면 THEN 시스템은 사용자에게 Gmail 권한 승인을 요청해야 합니다.
6. WHEN 이메일 발송이 성공하면 THEN 시스템은 발송된 메시지 ID와 발송 시간을 반환해야 합니다.
7. WHEN 이메일 발송이 실패하면 THEN 시스템은 500 에러를 반환하고 오류 메시지를 표시해야 합니다.

### 요구사항 6: 인원 관리

**사용자 스토리:** 사용자로서, 프로젝트 사용자 목록과 외부 연락처를 관리하여 일정 공유 시 쉽게 선택하고 싶습니다.

#### 수락 기준 (Acceptance Criteria)

1. WHEN 사용자가 인원 관리 페이지에 접근하면 THEN 시스템은 프로젝트 등록 사용자 목록을 표시해야 합니다.
2. WHEN 프로젝트 사용자 목록이 조회되면 THEN 각 사용자의 이메일, 이름, 프로필 사진, 가입일이 표시되어야 합니다.
3. WHEN 사용자가 외부 연락처 목록을 조회하면 THEN 시스템은 해당 사용자가 등록한 외부 연락처만 반환해야 합니다.
4. WHEN 사용자가 외부 연락처를 추가하면 THEN 시스템은 이름과 이메일을 필수로 요구해야 합니다.
5. WHEN 외부 연락처가 추가되면 THEN 시스템은 이름, 이메일, 전화번호, 회사, 직책, 메모를 저장해야 합니다.
6. WHEN 사용자가 외부 연락처를 삭제하면 THEN 시스템은 해당 사용자가 소유한 연락처만 삭제해야 합니다.
7. WHEN 다른 사용자의 외부 연락처를 삭제하려고 하면 THEN 시스템은 404 에러를 반환하고 접근을 거부해야 합니다.

### 요구사항 7: Google 토큰 관리

**사용자 스토리:** 사용자로서, Google 서비스(Calendar, Gmail) 접근 권한이 자동으로 관리되어 매번 재인증하지 않고 싶습니다.

#### 수락 기준 (Acceptance Criteria)

1. WHEN 사용자가 Google OAuth 인증을 완료하면 THEN 시스템은 액세스 토큰과 리프레시 토큰을 데이터베이스에 저장해야 합니다.
2. WHEN Google API 호출 전에 THEN 시스템은 토큰의 만료 시간을 확인해야 합니다.
3. WHEN 액세스 토큰이 만료되었으면 THEN 시스템은 리프레시 토큰을 사용하여 자동으로 새 액세스 토큰을 발급받아야 합니다.
4. WHEN 토큰이 갱신되면 THEN 시스템은 새로운 토큰 정보를 데이터베이스에 업데이트해야 합니다.
5. WHEN 리프레시 토큰도 만료되었으면 THEN 시스템은 사용자에게 재인증을 요청해야 합니다.
6. WHEN 사용자가 토큰 상태를 조회하면 THEN 시스템은 토큰 존재 여부와 만료 상태를 반환해야 합니다.

### 요구사항 8: 사용자 인터페이스

**사용자 스토리:** 사용자로서, 직관적이고 반응형인 웹 인터페이스를 통해 모든 기능을 쉽게 사용하고 싶습니다.

#### 수락 기준 (Acceptance Criteria)

1. WHEN 사용자가 로그인 페이지에 접속하면 THEN 시스템은 깔끔한 로그인 UI와 Google 로그인 버튼을 표시해야 합니다.
2. WHEN 사용자가 대시보드에 접근하면 THEN 시스템은 사이드바, 헤더, 메인 콘텐츠 영역을 포함한 레이아웃을 표시해야 합니다.
3. WHEN 사용자가 사이드바 메뉴를 클릭하면 THEN 시스템은 해당 섹션(통화 분석, 일정 관리, 인원 관리)을 표시해야 합니다.
4. WHEN 사용자가 통화 분석 섹션에 접근하면 THEN 시스템은 파일 업로드와 텍스트 입력 옵션을 제공해야 합니다.
5. WHEN 분석이 진행 중이면 THEN 시스템은 로딩 인디케이터를 표시해야 합니다.
6. WHEN 분석 결과가 표시되면 THEN 그룹 일정과 개인 일정이 구분되어 표시되어야 합니다.
7. WHEN 사용자가 일정을 수정하면 THEN 시스템은 인라인 편집 기능을 제공해야 합니다.
8. WHEN 화면 크기가 변경되면 THEN 시스템은 반응형으로 레이아웃을 조정해야 합니다.
9. WHEN 사용자가 모바일 기기에서 접속하면 THEN 시스템은 모바일 최적화된 UI를 표시해야 합니다.

### 요구사항 9: 오류 처리 및 보안

**사용자 스토리:** 사용자로서, 시스템이 안전하게 내 데이터를 보호하고 오류 발생 시 명확한 메시지를 제공받고 싶습니다.

#### 수락 기준 (Acceptance Criteria)

1. WHEN API 요청에 JWT 토큰이 없으면 THEN 시스템은 401 에러를 반환해야 합니다.
2. WHEN JWT 토큰이 유효하지 않으면 THEN 시스템은 401 에러와 함께 "유효하지 않은 토큰입니다" 메시지를 반환해야 합니다.
3. WHEN 사용자가 다른 사용자의 데이터에 접근하려고 하면 THEN 시스템은 404 에러를 반환해야 합니다.
4. WHEN 필수 필드가 누락되면 THEN 시스템은 400 에러와 함께 누락된 필드 정보를 반환해야 합니다.
5. WHEN 데이터베이스 오류가 발생하면 THEN 시스템은 500 에러를 반환하고 오류를 로깅해야 합니다.
6. WHEN 외부 API(OpenAI, Google) 호출이 실패하면 THEN 시스템은 적절한 에러 메시지를 반환해야 합니다.
7. WHEN 모든 API 요청이 처리되면 THEN 시스템은 요청 정보를 콘솔에 로깅해야 합니다.
8. WHEN 민감한 정보(토큰, 비밀키)가 로그에 포함되면 THEN 시스템은 해당 정보를 마스킹해야 합니다.

### 요구사항 10: 시스템 상태 모니터링

**사용자 스토리:** 개발자로서, 시스템의 각 서비스 상태를 확인하여 문제를 빠르게 파악하고 싶습니다.

#### 수락 기준 (Acceptance Criteria)

1. WHEN /health 엔드포인트에 접근하면 THEN 시스템은 전체 서비스 상태를 반환해야 합니다.
2. WHEN /api/analysis/health 엔드포인트에 접근하면 THEN 시스템은 분석 서비스와 GPT 연결 상태를 반환해야 합니다.
3. WHEN /api/schedules/health 엔드포인트에 접근하면 THEN 시스템은 일정 관리 서비스 상태를 반환해야 합니다.
4. WHEN /api/members/health 엔드포인트에 접근하면 THEN 시스템은 인원 관리 서비스 상태를 반환해야 합니다.
5. WHEN 서비스가 정상이면 THEN 시스템은 {"status": "healthy"} 응답을 반환해야 합니다.
